# README Medical Homepage Design

**Goal**

把 `README.md` 重构为面向医学用户的 GitHub 首页，使临床医生、PI 和医学研究合作者在首屏即可理解 `MedAutoScience` 是什么、解决什么问题、适合哪些研究场景，以及最终能形成什么样的研究产出。

同时保留对技术同事和 AI 执行者必要的详细技术说明，但不再让这些命令、实现层说明和内部术语主导首页叙事。

README 的外层称呼可以继续使用“医学自动科研平台”，但首页必须明确写出它的本质定义：

- 它是一个 `Agent-first, human-auditable` 的医学自动科研运行层
- 它不是要求医学研究者直接操作底层命令的工具箱
- 人类负责提出任务、提供数据、审阅结果和做关键决策
- Agent 负责调用平台接口推进研究与交付

## Primary Audience

- 临床医生
- PI
- 医学研究团队负责人

他们打开首页时最关心的问题是：

1. 这个项目是不是为医学研究而做，而不是通用 AI 自动化壳子
2. 它适合我手里的哪类数据和课题
3. 它能把课题推进到什么程度
4. 如果要让技术同事接手，下一步该去哪里看

## Secondary Audience

- 技术合作者
- AI 执行者
- 需要接入工作区或阅读内部实现的工程人员

这部分用户需要完整、明确、低歧义的技术入口，但不要求这些内容出现在首页主体前半段。

## Design Principles

### 1. 医学叙事优先

首页前半段按医学用户的阅读顺序组织：

- 这是什么平台
- 它解决什么问题
- 适合哪些数据与研究
- 最终会形成什么产出
- 平台如何工作

不在前半段展开 `controller`、`overlay`、`runtime`、`profile` 等工程实现术语。

### 2. 技术说明后移但不删除

现有详细技术说明继续保留，并通过清晰入口承接：

- 专门的 Agent 接口与运行说明文档
- `bootstrap/README.md`
- `controllers/README.md`
- `policies/*.md`

README 中新增一个明确标注的技术入口区，供技术同事和 AI 执行者按路径进入。

### 3. 首页不是命令手册

README 不再在主体部分直接铺开大量 CLI 命令。

首页最多保留一句技术入口引导，不承担部署细节、环境准备或内部控制器迁移说明。这些内容下沉到专门文档。

### 4. 单页双轨

同一个 `README.md` 同时服务两类用户，但通过结构分层避免互相打扰：

- 主路径：医学用户
- 次路径：技术同事 / AI 执行者

技术路径通过 GitHub 原生 `details` 折叠区承载，默认不干扰医学用户浏览。

## Information Architecture

README 重构后的信息顺序如下：

1. 顶部标题区
2. 首屏摘要
3. 主示意图
4. 这个项目适合谁
5. 这个项目解决什么问题
6. 当前已具备的关键能力
7. 你最终会得到什么
8. 适合哪些数据与研究类型
9. 平台如何工作
10. 默认优先研究场景
11. 给技术同事 / AI 执行者（折叠区）
12. 当前边界

### 首屏摘要约束

首屏必须在主示意图之前给出三个简短判断，确保医学用户在不滚动或少量滚动时就能完成初步理解：

- 适合谁
- 解决什么问题
- 最终得到什么

主示意图只负责辅助理解，不承担首屏核心判断信息，不允许再出现“大图在前、核心说明在后”的旧结构。

## Content Decisions

### 标题区

保留现有品牌视觉资产：

- `assets/branding/medautoscience-logo.svg`
- `assets/branding/medautoscience-hero.png`

标题区组成固定为：

1. Logo
2. 中文主标题
3. 一句面向医学用户的副标语
4. 首屏摘要
5. 主示意图

处理规则如下：

- 不保留当前偏 GitHub 仓库展示风格的 badge
- 不把英文副标语放在中文主标题上方或紧邻主标题的核心位置
- 如果保留英文表达，只能作为弱化的补充说明，不能抢占首屏主定位

副标语调整为更面向医学用户的表达，强调从专病数据到投稿交付，而不是强调通用“平台架构”。

### 首屏摘要

首屏摘要使用 3 条短句或 3 个简短条目，明确回答：

- 谁适合使用这个项目
- 它与通用自动科研工具的关键区别
- 它能把医学课题推进到什么程度

其中至少有一条要直接表达“Agent-first, human-auditable”的分工逻辑，但要用医学用户能读懂的话来写，不能写成纯工程口号。

这部分必须短、直、可快速扫读，不能写成抽象口号。

### 这个项目适合谁

新增面向角色的快速判断区，直接告诉医学用户：

- 如果你手里已有专病队列或临床数据库
- 如果你需要把课题推进到可投稿水平
- 如果你希望技术同事和 AI 协作而不是各自散做

那么该项目与其工作流相关。

### 这个项目解决什么问题

保留当前 README 中最有价值的核心判断，但改成面向医学用户的自然语言，不使用工程式分层说明。

重点保留以下主张：

- 不默认把整条自动科研流程机械跑完
- 优先判断一个方向是否值得继续投入
- 研究组织逻辑优先服从医学论文与临床读者，而不是通用 AI 论文范式
- 强调证据链完整性、发表门槛和止损换题

### 当前已具备的关键能力

新增一节独立的“当前已具备的关键能力”，用于回答“这个项目现在已经做到什么程度”，避免 README 变成只有愿景没有可信度的宣传页。

该节来自当前 README 中已实现能力的压缩重写，但表达方式改为医学用户能理解的结果语言，而不是工程模块枚举。

建议压缩为 4 到 6 条，例如：

- 已能把医学研究入口、策略门控与论文交付组织成闭环
- 已支持私有数据版本登记、公开数据扩展登记和影响评估
- 已把医学写作中的关键前验要求前移到流程中
- 已能导出最小投稿包并支持正式交付同步

这一节与“你最终会得到什么”分开：

- “当前已具备的关键能力”回答平台现在做到了什么
- “你最终会得到什么”回答用户在课题推进中可期待的交付

### 你最终会得到什么

新增结果导向的产出区，用医学用户更容易理解的话描述典型交付：

- 值得继续推进的研究方向
- 可追踪的数据资产与扩展线索
- 相对完整的分析结果与证据组织
- 面向投稿的稿件与交付材料

### 适合哪些数据与研究类型

这一节专门回答“我手里的数据和课题是否适合进入这个平台”，与后面的“默认优先研究场景”严格区分。

建议采用双列表或双列对照的短格式：

- 数据类型：临床结构化数据、影像、病理、多模态数据、组学数据、本地队列结合公开数据
- 课题类型：风险分层、外部验证、亚型重构、灰区分诊、机制扩展、临床窄任务智能体评测

这一节的职责是做输入适配判断，不负责表达平台策略优先级。

约束：

- 不写内部 archetype id
- 不展开技术接入方式
- 不与“默认优先研究场景”重复举例

### 平台如何工作

这一节改为面向医学用户的流程叙事，而不是内部组件命名表。

优先写成下面这种用户流程：

- 先判断课题是否值得继续投入
- 再组织私有数据与可用公开数据
- 再推进分析、扩展验证和证据组织
- 最后收敛为面向投稿的稿件与交付材料

内部组件名只作为次级标注出现，不作为该节主标题或主表格列名。

如果需要提到系统角色，保留高层业务角色即可：

- `MedAutoScience` 负责医学研究主控与门控
- `DeepScientist` 负责底层自动科研执行
- `Codex` 负责协调推进
- `ToolUniverse` 负责知识与功能扩展

不在该部分展开实现迁移状态或代码目录。

同时要明确人机分工：

- 人类定义问题、提供研究上下文、审阅结果并做关键判断
- Agent 负责调用运行层接口组织数据、推进分析和收敛交付

这一点是首页叙事核心，不应只出现在技术文档里。

### 默认优先研究场景

主页不再直接铺开英文 `archetype` 标识符表格。

该节使用医学问题场景来表达，例如：

- 临床风险分层与分类
- 数据驱动亚型重构
- 外部验证与模型更新
- 灰区分诊与追加检查分流
- 临床窄任务智能体评测
- 机制扩展与公开组学支持

如需查看内部 archetype 名称和策略细则，进入技术折叠区或相应 policy 文档。

这一节的职责是表达平台当前策略上优先支持、相对更稳定的研究推进方向，而不是再次说明可接收哪些数据输入。

### 技术入口与迁移映射

技术入口使用 `details` 折叠区，标题明确为“给技术同事 / AI 执行者”。

折叠区内按任务路径组织，不按文件名平铺：

- Agent 接入与运行接口：跳转专门的 Agent 使用说明文档
- 快速接入与部署：跳转 `bootstrap/README.md`
- 控制器与内部能力：跳转 `controllers/README.md`
- 平台规则与策略：跳转 `policies/data_asset_management.md`、`policies/study_archetypes.md`、`policies/research_route_bias_policy.md`

如果需要命令、profile、workspace 接入或 controller 迁移说明，都从这里进入，不再挤占首页主叙事。

为避免实现时技术内容迁移失控，README 改写时必须遵守以下映射：

| 当前 README 技术块 | 首页处理方式 | 目标落点 |
| --- | --- | --- |
| 数据资产层中的 CLI 命令清单 | 从首页正文移除，只保留一句技术 handoff | 专门的 Agent 接口文档 + `bootstrap/README.md` 与相关 `policies/*.md` |
| Agent mutation payload 示例 | 从首页正文移除，不在首页展示 JSON 示例 | 专门的 Agent 接口文档 |
| 最小部署中的命令与字段说明 | 从首页正文移除，只在折叠区给入口 | `bootstrap/README.md` |
| 平台核心组成中的内部命名说明 | 缩成业务角色说明，避免工程表格主导正文 | 首页“平台如何工作” + 折叠区补充 |
| 默认优先研究模式中的内部 archetype id | 改写为医学场景化表达 | 首页“默认优先研究场景” + `policies/study_archetypes.md` |
| 仓库文档列表 | 不再单独设置一个重复导航区 | 合并进技术折叠区 |

### 专门的 Agent 接口文档

为了保证 README 不承担教学手册职责，需要新增一份专门文档，承接 Agent 和技术同事真正需要的细节说明。

该文档至少包括：

- 运行层定位与人机分工
- 适合由 Agent 调用的稳定接口类型
- 数据资产相关 CLI 入口
- mutation payload 示例与使用边界
- 与 workspace/profile/bootstrap 的衔接路径

README 只提供这个文档的入口，不再直接展示命令清单和 JSON 负载。

因此，README 最终只保留一个技术 handoff 区，不再额外设置独立的“仓库文档导航”，避免重复导航和信息架构发散。

## Non-Goals

这次重构不做以下事情：

- 不改变项目真实能力边界
- 不新增产品承诺或不存在的研究模式
- 不删除现有技术文档
- 不把技术说明迁移为复杂文档站

## Success Criteria

重构后的 README 应满足：

1. 医学用户在不看命令的情况下，可以理解项目定位、适用场景和最终产出
2. 技术同事可以从首页一跳进入详细部署或实现说明
3. 现有技术资产保留，但不再压过首页主叙事
4. 首页首屏不依赖主示意图也能完成基本理解
5. 文案整体更像医学研究平台首页，而不是仓库内部开发备忘录
